# Workflow Format Specification

> Canonical contract between generators and converters. Both reference this spec as the authoritative format definition.

---

## Document Structure

Every workflow markdown file follows this structure:

```markdown
# [Platform] Workflows

> Auto-generated by [platform]-workflow-generator.
> Last updated: [date]
> Application: [app name]
> Base URL: [URL if known]

## Quick Reference

| # | Workflow | Priority | Auth | Steps |
|---|---------|----------|------|-------|
| 1 | User Registration | core | no | 7 |
| 2 | User Login | core | required | 5 |
| ... | ... | ... | ... | ... |

---

## Core Workflows

[Workflow 1 through N]

---

## Feature Workflows

[Workflow N+1 through M]

---

## Edge Case Workflows

[Workflow M+1 through end]

---

## Appendix: Application Map Summary

### Routes
[Summary table from exploration]

### Components
[Summary table from exploration]

### Data Model
[Summary table from exploration]
```

---

## Workflow Heading Format

Each workflow begins with a level-2 heading:

```markdown
## Workflow [N]: [Name]
```

Where `[N]` is a sequential integer starting at 1 and `[Name]` is a descriptive title.

---

## Metadata Comments

Every workflow has metadata comments immediately after the heading:

```markdown
## Workflow 1: User Login

<!-- auth: required -->
<!-- priority: core -->
<!-- estimated-steps: 5 -->
```

### Required Metadata

| Field | Values | Purpose |
|-------|--------|---------|
| `auth` | `required` \| `no` | Whether the workflow needs an authenticated session |
| `priority` | `core` \| `feature` \| `edge` | Workflow priority tier |
| `estimated-steps` | integer | Expected step count |

### Optional Metadata

| Field | Values | Purpose |
|-------|--------|---------|
| `deprecated` | `true` | Marks a workflow as deprecated (skipped by runner and converters) |

## Workflow Body

Each workflow contains three sections in order:

### Preconditions

```markdown
**Preconditions:**
- User is logged in as admin
- At least 1 project exists
```

### Steps

Numbered steps using this format:

```
N. Verb target — detail
```

Examples:
```markdown
1. Navigate to /dashboard — the main dashboard page
2. Click the "Create Project" button — opens the new project modal
3. Type "My Project" in the "Project Name" field — fill in the name
4. Verify the project appears in the list — confirms creation succeeded
```

### Postconditions

```markdown
**Postconditions:**
- A new project exists with the name "My Project"
- The user is on the project list page
```

## Recognized Verbs

All action steps must begin with one of these verbs:

| Verb | Description |
|------|-------------|
| Navigate | Go to a URL or page |
| Click | Click an element |
| Type | Enter text into a field |
| Select | Choose an option from a dropdown |
| Check | Check a checkbox |
| Uncheck | Uncheck a checkbox |
| Toggle | Toggle a switch or control |
| Upload | Upload a file |
| Download | Trigger a file download |
| Drag | Drag an element |
| Drop | Drop an element |
| Hover | Hover over an element |
| Scroll | Scroll to an element or position |
| Pause | Wait for a user-visible duration |
| Clear | Clear a field's contents |
| Submit | Submit a form |
| Dismiss | Dismiss a dialog, modal, or notification |
| Confirm | Confirm a dialog or action |
| Refresh | Reload the current page |
| Verify | Assert an expected outcome |

> **Note:** "Pause" is the workflow verb for user-visible wait steps. Internal timing mechanisms use snapshot polling via `playwright-cli` — these are distinct concepts.

---

## Verify Step Format

Verify steps assert expected outcomes. They should check specific, observable state:

```markdown
5. Verify the "Project Name" field displays "My Project" — name was saved
6. Verify the success toast shows "Changes saved" — confirmation message
7. Verify the URL contains "/projects/123" — navigated to project detail
```

### Recognized Assertion Types

| Type | Example |
|------|---------|
| Text content | Verify the heading displays "Dashboard" |
| Element state | Verify the "Submit" button is disabled |
| Element visibility | Verify the error message is visible |
| Element count | Verify there are 3 items in the list |
| URL | Verify the URL contains "/settings" |
| Absence | Verify the loading spinner is no longer visible |

### Visibility-Only vs Substantive

A Verify step is **visibility-only** if it only checks whether an element exists or is visible, without checking specific text, state, or values. Example: `Verify the dashboard is visible`. A **substantive** Verify checks specific content: `Verify the dashboard heading displays "Welcome, Admin"`. Max 30% of Verify steps should be visibility-only.

---

## Action Targets

Action targets must reference specific elements — not generic terms.

| Bad (generic) | Good (specific) |
|---------------|-----------------|
| Click the button | Click the "Submit" button |
| Type in the field | Type "admin@test.com" in the "Email" field |
| Check the checkbox | Check the "Remember me" checkbox |
| Select from the dropdown | Select "Admin" from the "Role" dropdown |

Targets should reference elements by: visible text, label, ARIA role, placeholder, or test-id.

## Multi-User Extensions

Multi-user workflow files add the following conventions.

### Persona Metadata

```markdown
<!-- personas: Admin, Host, Guest1 -->
```

Lists all personas that participate in the workflow.

### Persona Tags

Every step in a multi-user workflow must begin with a persona tag:

```markdown
1. [Admin] Navigate to /admin/users — admin dashboard
2. [Admin] Click the "Invite User" button — opens invite modal
3. [Host] Navigate to /inbox — check for invitation
4. [Host] Verify the invitation email from Admin is visible — cross-user check
```

### Cross-User Verification

Every mutation by one persona must be followed within N steps (default N=3) by a Verify step from a **different** persona. This ensures multi-user interactions are actually tested across browser contexts.

Every workflow must have at least 1 Verify step performed by a persona other than the one who performed the preceding mutation (User B Verify).

---

## Mobile Extensions

Mobile workflow files add the following metadata:

```markdown
<!-- viewport: mobile (393x852) -->
```

Mobile workflows follow the same format as desktop workflows but are authored and tested in a mobile viewport (393x852, iPhone 15 Pro equivalent).
